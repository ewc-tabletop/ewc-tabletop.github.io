Рассказать:

  • Общая функциональная схема приложения:
    • фронтенд, бэкенд-фасад, движок и набор правил
  • Конструктивная схема приложения:
    • для начала - самостоятельное приложение, поэтому движок будет подключен в код игры как зависимость, компонент развертывания будет по сути один
    • развитие - платформа для работы с любыми играми, тогда движок - это основной сервис, а отдельные фронтенды и наборы правил будут плагинами / микросервисами
  • Порядок разработки, варианты:
    • Подход Outside In Fake It Until You Make It
      • Минимальная реализация на каждом из этапов, исключающая инфраструктурные модули
      • Удобно разрабатывать через тестирование, при этом сначала будут написаны максимально общие приемочные тесты. Это про фрактальность TDD.
        • Очень большой вопрос по поводу того, каким должен быть порядок тестов? "Адвокат дьявола" поможет или будет мешать?
    • Подход Walking Skeleton
      • Минимально возможный вертикальный срез по всему приложению, включая инфраструктуру.
        • Мне этот вариант нравится меньше, потому что хоть он и снимает инфраструктурные риски, но заставляет принимать решения о составе модулей проекта на старте, когда еще неизвестны точные требования
